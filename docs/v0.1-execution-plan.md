# v0.1 Execution Plan

_Last updated: 2026-03-21 (UTC)._  
_Purpose: convert the roadmap into implementation-ready deliverables with acceptance criteria._

## TIS // Tactical Implementation Spec

### SITREP
- **Current State:** The main plan defines milestones, but not owner-ready acceptance criteria, decision gates, or operational outputs.
- **Objective:** Translate the roadmap into a fast, execution-grade v0.1 plan focused on Egypt first, then Lebanon/Syria, then Saudi/UAE support tracks.
- **Threat Assessment:** The biggest risk is overbuilding before the first backtest proves useful. The second risk is under-defining “done” and drifting into endless planning.

### BATTLE PLAN
- **OBJECTIVE:** Build only what is required to produce analyst-reviewable anomalies quickly.
- **CURRENT_STATE:** The core plan exists, but execution criteria do not.
- **ACTIONS:**
  1. Finish the shared scoring and district-selection artifacts first.
  2. Build Egypt as the primary v0.1 pilot.
  3. Add Lebanon/Syria corridor as the first movement-volatility expansion.
  4. Add Saudi/UAE only after Egypt backtest quality is acceptable.
- **VERIFICATION:** Every milestone must end with a specific output, acceptance test, and go/no-go decision.
- **CONTINGENCY:** If Egypt signal quality is poor after milestone 2, pause expansion and revisit lexicons, districts, and confound handling.

### QUALITY METRICS & SUCCESS VALIDATION
- **Metric 1:** Time from source fetch to analyst-reviewed anomaly card.
- **Metric 2:** Percentage of anomalies that survive second-source review.
- **Success Validation:** v0.1 is successful when Egypt produces a usable backtest and one live weekly review cycle without policy violations.

### SECURITY & COMPLIANCE GATES
- **Lint/Static:** Not applicable; documentation only.
- **CVE Scan:** Not applicable; no software changes.
- **Unsafe patterns avoided:** No restricted collection, no platform evasion, no person-level inference.

## 1) v0.1 Build Sequence

### Phase 0: Planning hardening (1-2 days)
**Goal:** remove ambiguity before engineering begins.

**Deliverables**
- confidence rubric approved,
- district-selection matrix approved,
- top-20 district shortlist,
- source-registry owner assignments.

**Acceptance criteria**
- analysts can score the same sample anomaly within ±10 points,
- the first 20 districts are named and paired with controls where possible,
- every v0.1 source has an assigned owner and cadence.

### Phase 1: Egypt baseline ingestion (2-4 days)
**Goal:** stand up the first hard-baseline pipeline.

**Inputs**
- UNHCR Egypt,
- IOM DTM Sudan,
- selected Cairo/Giza district list.

**Outputs**
- normalized event baseline table design,
- event timeline for Egypt pilot,
- district watchlist for Cairo/Giza.

**Acceptance criteria**
- Sudan-linked baseline events are loaded for the pilot time window,
- target districts and controls are frozen for the first backtest,
- source URLs and access timestamps are captured for every baseline record.

### Phase 2: Egypt market-proxy collection (3-5 days)
**Goal:** produce the first analyzable non-humanitarian signal set.

**Inputs**
- Google Places / OSM queries,
- public merchant pages,
- menu lexicon v1.

**Outputs**
- bakery/grocery/café seed dataset,
- menu-term extraction checklist,
- first anomaly candidates for analyst review.

**Acceptance criteria**
- at least two source families exist for each included pilot district,
- at least one district shows a reviewable candidate anomaly,
- no collection step violates the documented guardrails.

### Phase 3: Egypt backtest and analyst queue (2-3 days)
**Goal:** prove the pilot is worth scaling.

**Outputs**
- scored anomaly cards,
- analyst review worksheet,
- go/no-go summary for expansion.

**Acceptance criteria**
- at least one anomaly is supported by one hard baseline plus one market proxy,
- each anomaly has a confidence score and confound note,
- the analyst queue can be run weekly with the available evidence.

### Phase 4: Lebanon / Syria corridor expansion (3-5 days)
**Goal:** test volatility and signal disappearance, not just emergence.

**Outputs**
- cross-border event timeline,
- district pack for Akkar / Tripoli / Bekaa / Beirut periphery,
- bakery-open-close review list.

**Acceptance criteria**
- event overlays are aligned to flash-update windows,
- at least one example of appearance or disappearance is scored,
- controls are used to rule out generic seasonality where possible.

### Phase 5: Saudi and UAE support tracks (3-5 days)
**Goal:** add retail- and premiumization-oriented comparators without distracting from migration-linked core work.

**Outputs**
- Saudi retail chain monitoring shortlist,
- UAE premiumization watchlist,
- explicit rule separating premiumization signals from migration-linked signals.

**Acceptance criteria**
- Saudi work focuses on measurable assortment and chain signals,
- UAE work is capped as supporting evidence unless harder baselines exist,
- neither track delays the Egypt or Lebanon review cycle.

## 2) Weekly Review Loop

Each weekly cycle should produce:
- refreshed baseline event notes,
- refreshed place or merchant snapshots,
- anomaly score sheet,
- analyst review notes,
- deferred collection actions for weak cases.

## 3) Go / No-Go Gates

### Expand from Egypt only if:
- anomaly precision is acceptable,
- confidence scoring is stable,
- review time is manageable,
- collection remains compliant.

### Pause and rework if:
- most cases remain `Unconfirmed`,
- confounds dominate the explanations,
- district coverage is too thin,
- or analyst workload becomes unsustainable.

## 4) Minimal v0.1 Output Pack

By the end of v0.1, the repo should contain:
- source registry,
- confidence rubric,
- district-selection matrix,
- event timeline template,
- anomaly review worksheet,
- pilot execution summary.
